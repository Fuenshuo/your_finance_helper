**目标**: 将现有功能强大的模块化系统，无缝迭代至PRD V2.4所定义的、以“三层财务模型”为核心的一体化智能财务引擎。

## 第一部分：核心架构升级：从“功能聚合”到“三层联动”
这是本次迭代的**顶层设计**。我们需要将您现有的10个独立模块，重新组织进PRD V2.4定义的**“信息维护 → 财务计划 → 交易流水”**这一核心逻辑链中。这主要是概念和入口的整合，而非功能的废弃。

| 原有模块 | 迭代后归属 | 核心变化 |
| --- | --- | --- |
| 工资管理、税务计算 | **财务计划** (收入计划) | 从独立系统，变为创建“工资收入计划”时的深度配置项。 |
| 资产管理、账户管理 | **家庭信息维护** | 合并为统一的静态资产管理入口。 |
| 预算管理、贷款计算 | **财务计划** (支出计划) | 预算变为约束型计划；贷款变为负债资产，并自动生成还款计划。 |
| 交易管理 | **交易流水** | 保持核心功能，增强与“计划”的关联。 |
| 历史追溯、性能、开发 | **系统底层/设置** | 功能保留，入口移至设置或后台。 |


## 第二部分：功能模块迭代详解
### **迭代模块一：整合为「家庭信息维护」**
此模块的目标是统一管理所有“静态”的财务信息。

#### **1.1 「账户管理系统」 → 升级为「钱包管理」**
+ **差异点**: 您当前的“账户”概念与PRD V2.4的“钱包”完全一致，仅需统一命名。
+ **迭代任务**:
    - 在UI层面，将所有“账户”字样统一为“**钱包**”，强化其作为“资金实际载体”的定位。
    - 将原独立的**【模块五：账户管理系统】**的入口，整合进新的**【家庭信息维护】**模块中。功能完全保留。

#### **1.2 「资产管理系统」 → 融入并深化**
+ **差异点**: 您当前的资产管理是静态的，只记录账面价值。V2.4的核心是引入**动态估值**。
+ **迭代任务**:
    1. **入口整合**: 将原**【模块二：资产管理系统】**入口，同样整合进**【家庭信息维护】**。
    2. **核心功能新增：专业估值体系**
        * **UI/UX设计**:
            + 在用户添加“固定资产”或“投资理财”时，在原有的录入流程后，增加一步“**设置估值方式**”。
            + 提供一个选择器，让用户根据资产类型（房产、汽车、大宗商品、收藏品）选择不同的估值模型。
        * **房产估值**:
            + 提供两个选项卡：“**手动更新**”和“**自动增长**”。
            + 选择“自动增长”时，提供输入框让用户填写预估的`年化增长率(%)`。
        * **汽车估值**:
            + 提供“**自动折旧**”选项。当用户选择此项并输入品牌型号后，应用可提示一个行业参考折旧率（如`15%`），用户可修改。
            + 系统需在资产详情页展示一个由`购入价`和`当前估值`组成的价值变化曲线图。
        * **大宗商品估值**:
            + 流程类似汽车，但折旧模型更简化，用户只需选择`使用年限`（如3年、5年）。
        * **资产详情页改造**:
            + 在每个非流动性资产的详情页，醒目地展示其`购入原价`和`当前估值`，并提供一个“**更新估值**”的按钮，允许用户随时手动修正。

### **迭代模块二：重构为「财务计划」**
这是本次迭代的**核心**，将多个独立模块的能力，统一到“计划”这个生活化概念之下。

#### **2.1 「工资/税务系统」 → 整合为「周期性收入计划」**
+ **差异点**: 您当前的工资和税务系统非常强大但独立。我们需要将其能力作为创建“收入计划”时的一个**高级选项**。
+ **迭代任务**:
    1. 创建一个全新的**【财务计划】**模块。
    2. 在其中提供“**新建计划**”按钮，类型可选“收入计划”或“支出计划”。
    3. 当用户创建“**收入计划**”时：
        * 提供一个“**计划模板**”选择，选项为“**普通收入**”和“**详细工资**”。
        * 选择“**普通收入**”：只需填写金额、周期、目标钱包等简单信息。
        * 选择“**详细工资**”：则加载您现有**【模块一：工资管理系统】**的完整界面，让用户进行基本工资、奖金、补贴、五险一金、专项附加扣除等详细设置。**【模块六：税务计算系统】**的计算能力在此处被调用。
        * 保存后，无论模板如何，最终都生成一条“收入计划”。

#### **2.2 「预算/贷款系统」 → 整合为「支出计划」**
+ **差异点**: 预算和还贷目前是独立功能，新架构中它们都是“支出计划”的不同表现形式。
+ **迭代任务**:
    1. 当用户创建“**支出计划**”时：
        * 提供计划类型选择：“**周期性支出**”和“**预算计划(约束型)**”。
    2. **创建“周期性支出”计划**:
        * **UI/UX**: 类似收入计划，填写名称（如房租）、金额、周期、支付钱包。
        * **与贷款联动**:
            + 原**【模块八：贷款计算系统】**的功能，应在**【家庭信息维护】->【资产管理】**中添加“负债”时被调用。
            + 当用户成功添加一笔贷款后，系统应**自动弹窗**提示：“**已为您创建了一条‘[贷款名称]还款’支出计划，每月将自动记录还款，是否立即查看？**”
            + 用户可在“财务计划”模块中看到这条自动生成的计划。
    3. **创建“预算计划(约束型)”**:
        * **UI/UX**: 加载您现有**【模块三：预算管理系统】**的界面。
        * **核心逻辑改造**:
            + 废弃“零基预算”中“将总收入分配”的操作，因为“收入”已经由“收入计划”管理。
            + 保留“信封预算”的界面，但改变其话术和底层逻辑。用户不再是“往信封里存钱”，而是“**为‘餐饮’这个分类设置每月1000元的消费上限**”。
            + 这个计划本身**不产生交易**，只在发生实际消费交易时，作为一条**校验规则**被触发。

### **迭代模块三：增强「交易流水」**
#### **3.1 「交易管理系统」 → 融入并增强**
+ **差异点**: 您当前的交易系统功能完整，但与“计划”是脱节的。
+ **迭代任务**:
    1. **入口更名**: 将**【模块四：交易管理系统】**的入口，更名为**【交易流水】**，作为一级导航。
    2. **核心功能新增：计划关联**
        * **UI/UX**: 在交易流水的列表和详情页，对于由“财务计划”自动生成的交易，需要增加一个明显的**标签或小图标**（如“自动执行”）。
        * 点击该标签或在详情页，应显示“**来自您的‘工资计划’**”，并提供一个链接，点击可跳转到该计划的详情页进行编辑。
    3. **核心功能新增：智能建议**
        * **逻辑**: 当用户**连续第二次**手动记录一笔信息高度相似（分类、金额范围、对方账户等）的交易时，系统在保存成功后应弹窗提示。
        * **话术**: “**我们注意到您已连续两个月记录了‘房租’支出，需要为您创建一个每月自动执行的支出计划吗？**”
        * **操作**: 提供“创建计划”和“不用了”两个按钮。点击“创建计划”则预填写好信息，引导用户快速保存。

通过以上迭代，您现有系统的强大功能不仅得到了完整保留，还被一套更先进、更符合用户直觉的架构有机地串联了起来，最终将实现PRD V2.4所描绘的宏伟蓝图。
